# AGENTS.md

本文件只记录 Codex / AI 助手在本项目中的工作规则。项目长期背景请阅读 `docs/MASTER_CONTEXT.md`，产品上下文请阅读 `docs/PRODUCT_CONTEXT.md`。

## AI 工作规则

- 输出要区分：已知事实、判断、待验证假设、下一步行动。
- 不要泛泛而谈，要给建设性意见。
- 如果用户想法分散、逃避、过度宏大或有明显漏洞，要直接指出，并给出可执行收敛方式。
- 不要一次性生成巨大方案，要一步一步指导。
- 用户是新手开发者，需求描述可能不专业；AI 要先用产品经理思维理解业务需求，再转成技术方案。
- 如果关键信息不足，只问 1-3 个最关键问题。
- 每次任务结束后，说明新增、修改、删除了哪些文件。
- 每次任务结束后，更新 `docs/CURRENT_TASK_STATE.md`。

## 代码修改流程

执行任何代码修改前，必须先完成以下步骤：

1. 阅读 `README.md`、`AGENTS.md`、`docs/MASTER_CONTEXT.md`、`docs/TASKS.md`。
2. 检查当前 git 状态，避免覆盖用户已有改动。
3. 复述理解的目标、约束和影响范围。
4. 说明计划、影响范围、风险和测试方式。
5. 等待用户确认，除非用户已经明确要求直接执行且改动范围很小。

执行原则：

- 优先最小改动，不要为了小问题大范围重写模块。
- 不允许未经确认进行大规模重构。
- 不允许未经确认删除文件。
- 不允许未经确认新增大型依赖。
- 新增任何依赖前，必须说明原因、替代方案和维护成本。
- 重要修改后必须说明如何回滚。
- 如果修改影响启动方式、目录结构、环境变量、接口或用户使用方式，必须同步更新 `README.md` 或相关 `docs/` 文档。

## 禁止事项

- 不允许修改生产环境配置或密钥。
- 不得修改 `.env`、`.env.local`、密钥、token、数据库连接串。
- 如必须说明配置，只能修改 `.env.example`，且不能写入真实密钥。
- 不得未经确认执行数据库迁移、清空数据、重置数据库、删除表、覆盖数据等操作。
- 不允许运行破坏性命令，除非用户明确要求并确认影响。
- 不允许假装测试成功。
- 不允许编造项目状态、技术栈、运行方式、用户数据或业务结论。

## 测试要求

- 修改代码后必须说明验证方式。
- 能运行测试时，应运行与改动范围匹配的最小测试。
- 测试失败时必须如实说明失败原因、失败命令和影响。
- 如果无法运行测试，必须说明原因，并给出可替代的人工验证方式。
- 文档类修改至少需要检查目标文件是否存在、内容是否覆盖用户要求。

## 不确定信息处理方式

- 不确定的信息必须标记为“待确认”。
- 不能为了让文档完整而编造细节。
- 从用户输入得到的信息标记为“已知事实”。
- 基于当前项目结构、文件检查或命令输出得到的信息，可以标记为“当前真实结构”或“本轮检查结果”。
- 推断性内容必须标记为“判断”或“待验证假设”。
